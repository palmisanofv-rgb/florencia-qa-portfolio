# Tool Selection Matrix

A QA Lead's job includes choosing tools for a team's actual skill mix, budget, and CI constraints — not defaulting to whatever the lead personally prefers. This matrix documents *why* each tool was picked for each project in this portfolio, as if each were a real team decision.

| Tool | Used in | Chosen because | Trade-off accepted |
|------|---------|-----------------|---------------------|
| **Playwright + TypeScript** | [01](../01-ecommerce-automationexercise) | Fastest modern option for teams already writing TS/JS elsewhere in the stack; built-in auto-waiting reduces flakiness; native API-testing support (used the same tool for UI *and* API in Project 01) | Smaller long-term hiring pool than Selenium; younger ecosystem for enterprise plugins |
| **Selenium + Java + TestNG** | [02](../02-ecommerce-saucedemo-selenium-java), [04 (Python variant)](../04-banking-parabank-selenium-python) | Still the largest hiring pool and most enterprise-proven WebDriver implementation; TestNG's data providers and parallel execution suit large regression suites | More boilerplate than Playwright; explicit waits required (no built-in auto-wait) |
| **Katalon Studio** | [03](../03-healthcare-cura-katalon) | Low-code authoring lets manual-leaning testers contribute automation directly (object repository + keyword-driven scripts), without requiring every tester to be a strong developer | Vendor lock-in risk; less flexible than raw code for complex logic |
| **WebdriverIO + JavaScript** | [08](../08-technique-lab-the-internet-webdriverio) | Cleanest API in the WebDriver family for edge-case interactions (shadow DOM piercing, native dialogs) — chosen here specifically for technique coverage, not as a team default | Smaller community than Selenium/Playwright for troubleshooting |
| **Appium + Java** | [09](../09-mobile-appium-sampleapp) | Same WebDriver protocol as Selenium — a Selenium-literate team can pick up mobile automation without learning an entirely new paradigm | Requires real device/emulator infrastructure to run in CI, which is the actual cost center for mobile automation, not the scripting itself |
| **Postman/Newman** | [01](../01-ecommerce-automationexercise), [05](../05-media-tvmaze-api-postman), [07](../07-saas-dummyjson-api-postman) | Lowest barrier to entry for API testing; collections are readable/maintainable by less code-heavy team members; Newman makes it CI-portable | Less expressive than writing API tests in the same language as the rest of the suite (harder to share fixtures/helpers with UI code) |
| **Apache JMeter** | [06](../06-travel-blazedemo-jmeter) | Industry-standard for performance testing; GUI authoring lowers the barrier for a QA team without dedicated performance engineers | Steep learning curve past basic scripts; XML-based `.jmx` is hard to code-review meaningfully in a PR |
| **SQL** | [04](../04-banking-parabank-selenium-python), [10](../10-database-sql-etl-validation) | The only way to catch defects that live *below* the UI/API layer — data integrity, ETL correctness, referential integrity | Requires (authorized) database access most QA teams don't have by default — a process/access decision for a QA Lead to negotiate, not a tooling one |

## Decision framework this reflects

When a team lead is picking a tool, the real question isn't "which tool is best" — it's: *who on the team will maintain this, what does the CI budget allow, and what failure mode does this specific layer need to catch that other layers can't?* This portfolio deliberately uses 7 different tools across 10 projects to demonstrate range, but a real team wouldn't run 7 tools at once without a reason — the reason, here, is that each project intentionally isolates the scenario where that specific tool is the right call.
